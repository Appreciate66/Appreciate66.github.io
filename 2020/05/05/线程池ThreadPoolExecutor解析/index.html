线程池ThreadPoolExecutor解析

发表于 2020-05-05 | 分类于 JUC | | 阅读数
字数统计: 6.3k 字 | 阅读时长 ≈ 25 分钟

​ 线程池在平时的开发中想必大家都用到过，使用的方式也千奇百怪。那么在写这篇文章之前，需要要先抛出一些问题：什么是线程池？为什么需要有线程池的存在？线程池的正确用法是什么？线程池内部工作原理是怎样的？带着这些问题去学习线程池，看线程池的源码，我想这样对我们的收获才是最大的。不仅如此，还能学习到Doug Lea大神是如何设计并实现线程池的（如果不知道Doug Lea是谁，请google一下#狗头）。

线程池概念

​ 在学习任何一个东西之前，了解基本概念的定义是至关重要的。大脑中需要先有宏观的概念，从宏观到微观，这样理解不仅全面，还会加深记忆。

####为什么要有线程池？

​ 在生产环境中，我们不可能每执行一个任务就开一个线程，这样做有一些明显的缺陷，尤其是在需要创建大量线程的情况下。例如线程的创建与销毁会浪费大量的资源；活跃的线程会消费系统资源，尤其是内存，如果线程的数量超过可用处理器的数量，那么就会有空闲线程闲置，闲置的线程也会占用内存。

​ 显然在这样的情况下，如果能够复用线程，减少线程的创建数量，让CPU数与线程数达到某种合理的平衡，这样的话即能够提升系统的性能，又能减少创建，销毁线程带来的资源开销。于是就有了线程池的概念。

线程池的优点

  • 通过重用现有的线程而不是创建新线程，可以在处理多个请求时分摊在线程创建和销毁过程中产生的巨大开销。
  • 当请求到达时，工作线程通常已经存在，不会由于等待创建线程而延迟任务的执行，从而提高了响应性。
  • 通过适当调整线程池的大小，可以让线程数刚好够以便使处理器保持忙碌状态，同时还可以防止过多线程相互竞争资源使内存耗尽。

线程池怎么用？

​ 线程池怎么用包含了两个方面：一是如何创建线程池并配置正确的线程池参数；二是该在哪种场景下使用哪一种线程池。

​ java.util.concurrent包下提供了一种灵活的线程池实现作为Executor框架的一部分。

​ Executor是一个接口，Runnable表示任务，接口的行为很简单：执行任务，同时将任务的提交与执行过程解耦开。

####线程池的配置参数

​ 在对如何创建并使用线程池之前需要先花一些篇幅讲解下线程池的配置参数，正确配置参数是非常重要的，线程池玩的溜不溜，要看参数配的好不好，配不好，线上OOM的情况会经常出现，配的好的话，任务执行的效率会大大的提高。

​ 无论你采用哪种方式创建线程池，最终都会进入到上面的构造方法中，而这些参数的作用非常重要并且也是面试中面试官经常会问到的问题，我们来看看这些参数意义跟作用是什么：

  • corePoolSize：核心线程数

  • maximumPoolSize：线程池允许创建的最大线程数

  • keepAliveTime：空闲线程的存活时间

  • workQueue：存放任务的任务队列，有界队列（ArrayBlockingQueue） 和无界队列（LinkedBlockingQueue）

  • threadFactory：线程工厂，Executor通过threadFactory来创建线程，可以设置线程名称

  • handler：拒绝策略，当任务队列满时或线程池中线程达到最大线程数时线程池执行的策略

    看到参数介绍以后我想应该会有一些疑问：

    （1）如何设置核心线程数与最大线程数？

    （2）线程池中的线程是在什么情况下创建？

    （3）该采用无界队列还是有界队列？如果采用有界队列，任务队列如果满了怎么办？采用无界队列会有什么问题？

    （4）线程池提供的拒绝策略一共有几种，可以自己实现拒绝策略吗？

    ​ 好像问题比较多，但是没有关系，先卖个关子，这些问题会在下文的线程池源码分析中得到答案。我们先来看看线程池怎么创建的。

####Executors创建线程池

最简单创建线程池的方式就是通过Executors中的静态工厂方法来创建,一共有4种（newWorkStealingPool本质上不算线程池的扩展，本文不做解析）。

​ newFixedThreadPool将创建固定长度的线程池，每当提交一个任务时就创建一个线程，任务队列采用无界队列LinkedBlockingQueue。

​ newSingleThreadExecutor的corePoolSize跟maximumPoolSize都是1，所以它是一个单线程的Executor，它创建单个线程来执行任务，如果这个线程异常结束，会创建另一个线程来替代它。任务队列采用无界队列LinkedBlockingQueue。

​ newCachedThreadPool会创建一个可缓存的线程池，可以看到maximumPoolSize=Integer.MAX_VALUE；这意味着线程池的规模不限，有任务进来就创建线程，当空闲线程空闲时间超过60秒后就会被回收。

​ ScheduledThreadPoolExecutor创建了一个固定长度的线程池，而且以延迟或定时的方式来执行任务。

​ 而在阿里巴巴开发手册里明确指出了在生产环境不允许通过Executors的方式来创建线程池，而是要通过ThreadPoolExecutor的方法来创建，为什么？FixedThreadPool 和 SingleThreadPool允许的请求队列长度为 Integer.MAX_VALUE，可能会堆积大量的请求，从而导致 OOM。CachedThreadPool：允许的创建线程数量为 Integer.MAX_VALUE，可能会创建大量的线程，从而导致 OOM。

ThreadPoolExecutor创建线程池

​ ThreadPoolExecutor是一种较为灵活的，稳定的线程池，允许进行各种定制。通常在生产环境我们应该根据实际的业务需要与产线的机器配置来动态设置线程池的参数。

#####核心线程数与最大线程数设置

​ corePoolSize与maximumPoolSize的设置应参考具体的业务场景。笔者认为应当从下面几个方面来考虑：

  • 任务执行类型：计算密集，I/O密集，混合型

  • 响应时间敏感：长，短

  • 执行时间：长，短

计算密集型任务，在拥有N个cpu处理的系统上，线程池的大小因为N+1；而I/O密集型任务应尽可能配置多的线程，因为I/O操作并不会占用CPU，理论上线程池的大小应为2N+1,而针对混合型任务，则需要考虑是否能对任务进行拆分。

响应时间敏感指的是用户在页面单击按钮后需要多少的时间才能获取结果，这个时候如果将多个运行时间较长的任务提交到一个只包含少量线程的线程池中，那么该服务的响应性会降低，所以应该适当调大线程的线程数。

执行时间较短的任务，线程池线程的数量不应多大，这样可避免线程间上下文切换。

（1）高并发、任务执行时间短的业务怎样使用线程池？

​ 高并发、任务执行时间短的业务，线程池线程数可以设置为CPU核数+1，减少线程上下文的切换

（2）并发不高、任务执行时间长的业务怎样使用线程池？

并发不高、任务执行时间长的业务要区分开看：
　　a）假如是业务时间长集中在IO操作上，也就是IO密集型的任务，因为IO操作并不占用CPU，所以不要让所有的CPU闲下来，可以适当加大线程池中的线程数目，让CPU处理更多的业务
　　b）假如是业务时间长集中在计算操作上，也就是计算密集型任务，这个就没办法了，线程池线程数可以设置为CPU核数+1，减少线程上下文的切换
（3）并发高、业务执行时间长的业务怎样使用线程池？

并发高、业务执行时间长，解决这种类型任务的关键不在于线程池而在于整体架构设计，看看这些业务里面某些数据是否能做缓存是第一步，增加服务器是第二步，至于线程池的设置，设置参考（2）。最后，业务执行时间长的问题，也可能需要分析一下，看看能不能使用中间件对任务进行拆分和解耦。

任务队列

​ 无界队列在高并发情况下，会无限制的向队列里添加任务从而耗尽资源导致OOM使整个服务不可用。

​ 而使用有界队列时，队列的大小与线程池的大小必须一起调节。如果线程池较小而队列较大，那么有助于减少内存使用量，降低CPU的使用率，同时还可以减少上下文切换，但吞吐量可能会受到限制。而另一个问题则是当有界队列满时，对新任务的提交采用哪种饱和策略？

为你的线程打上标记

​ 线程池创建线程是通过ThreadFactory来创建的,我们可以对线程设置名称等，这里推荐使用google的guava来创建。

饱和策略

​ 当有界队列填满后，饱和策略开始发挥作用，JDK提供了4种不同的RejectedExecutionHandler实现，AbortPolicy，DiscardPolicy,DiscardOldestPolicy,CallerRunsPolicy,我们也可以自己实现RejectedExecutionHandler来定制属于自己的拒绝策略。

AbortPolicy：默认的饱和策略，当队列满时，该策略抛出RejectedExecutionException异常，我们可以捕捉这个异常来编写处理代码。

DiscardPolicy：偷偷的把任务扔了，我也不告诉你。

DiscardOldestPolicy：抛弃队列中下一个将被执的任务，然后执行新任务

CallerRunsPolicy：新任务回退给调用者，从而降低新任务的流量。

线程池使用的一些思考与借鉴

​ 由上可知，线程池参数设置并没有一套统一的标准，既然没有标准，那就不应该在代码中配置写，应将修改线程池参数的成本降下来，那么可以将参数的配置放到配置平台上去做，同时增加线程池核心状态的监控，对某些核心参数进行阈值设置，超过阈值报警或者发邮件，这样可以在第一时间出现问题通知到开发人员，从而加速故障恢复的速度。

  • 增加线程池监控

ThreadPoolExecutor源码分析

​ ThreadPoolExecutor的源码分析放到了最后，在了解线程池基本的概念与常规配置后，我们通过对线程池的源码分析来了解线程池的具体执行流程时怎么样的。

线程池的状态

​ 线程池的状态是通过AtomicInteger原子类型来表示：

​ workerCount：线程池中有效的线程数：(2^29)-1，大概5亿多

​ runState：线程池的运行状态

​ RUNNING：接受新任务并处理排队的任务

​ SHUTDOWN：不接受新的任务了，但是队列中的任务还是会处理

​ STOP：不接受新的任务，不处理队列中的任务，中止正在处理的任务

​ TIDYING：所有任务终止，工作线程数为0，线程池状态会过渡到TIDYING状态，并执行钩子函数terminated()

​ TERMINATED：钩子函数terminated()执行后的状态

线程池状态的过渡流程：

​ RUNNING -> SHUTDOWN：调用shutdown()方法后，或者隐式的在finalize()方法里调用

​ (RUNNING or SHUTDOWN) -> STOP：调用shutdown()方法后

​ SHUTDOWN -> TIDYING：当线程池或者队列为空时

​ STOP -> TIDYING：当线程池为空时

​ TIDYING -> TERMINATED：当钩子函数terminated()执行完毕后

####核心方法execute(Runnable command)

​ 从源码中可以看到线程池中创建线程的时机有两个：

​ 1.当前线程池中线程数小于核心线程数时，新的任务会创建新的线程。

​ 2.当线程池中线程数大于核心线程数并且有界队列满时，这时新任务会创建线程直到线程数达到最大线程数。

​ 备注：如果队列设置成无界队列，那么当线程池中线程数大于核心线程数时，并不会创建新的线程，而是无脑把任务扔到队列中

​ 执行饱和策略的场景也有两个：

​ 1.线程池中的线程数达到了核心线程数，任务队列中有新的任务了，但是同时线程池关闭了，这时再添加任务则会执行饱和策略。

​ 2.线程池中队列已满并且线程数达到最大线程数，那么这时再次添加任务时会执行饱和策略。

真正干活的：Worker

内部类Worker

​ execute(Runnable command)方法中频繁出现了addWorker(command, boolean)这个方法，在阅读这个方法前，先看一下Worker这个内部类，Worker继承了AQS框架并实现了Runnable接口，内部有一个初始化的任务firstTask，thread是通过构造方法里ThreadFactory创建的，这里面firstTask很有意思，firstTask代表如果在线程创建前有值，那么线程在启动后则会直接执行这个任务，否则就要去队列里拉任务执行了。

​ Worker是通过继承AQS，使用AQS来实现独占锁这个功能。没有使用可重入锁ReentrantLock，而是使用AQS，为的就是实现不可重入的特性去反应线程现在的执行状态。

1.lock方法一旦获取了独占锁，表示当前线程正在执行任务中。

2.如果正在执行任务，则不应该中断线程。

3.如果该线程现在不是独占锁的状态，也就是空闲的状态，说明它没有在处理任务，这时可以对该线程进行中断。

4.线程池在执行shutdown方法或tryTerminate方法时会调用interruptIdleWorkers方法来中断空闲的线程，interruptIdleWorkers方法会使用tryLock方法来判断线程池中的线程是否是空闲状态；如果线程是空闲状态则可以安全回收。

添加任务:addWorker(Runnable firstTask, boolean core)
执行任务：runWorker(Worker w)

​ worker 中的线程 start 后，其run方法会调用runWorker方法

​ 线程池中线程的销毁依赖JVM自动的回收，线程池做的工作是根据当前线程池的状态维护一定数量的线程引用，防止这部分线程被JVM回收，当线程池决定哪些线程需要回收时，只需要将其引用消除即可。Worker被创建出来后，就会不断地进行轮询，然后获取任务去执行，核心线程可以无限等待获取任务，非核心线程要限时获取任务。当Worker无法获取到任务，也就是获取的任务为空时，循环会结束，Worker会主动消除自身在线程池内的引用。核心线程是否要回收通过：allowCoreThreadTimeOut（boolean）来设置。

获取任务：getTask()

1.由于调用setMaximumPoolSize方法从而导致worker数大于MaximumPoolSize
2.线程池状态==stopped.
3.线程池状态==shutdown并且队列为空
4.worker等待任务超时，超时的worker将终止
